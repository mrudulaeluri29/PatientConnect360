## Phase 6 — Dashboard Icon Language And Emoji Removal

### Goal
Remove emoji and ad-hoc symbol glyphs from all dashboard experiences and replace them with a clean, familiar, signage-first icon system.

This phase must:
- remove emoji from patient, caregiver, clinician, and admin dashboards
- replace raw emoji/symbol strings with sleek icon components wherever possible
- use a single recognizable icon language across shared dashboard modules
- improve familiarity and wayfinding through consistent signage
- preserve all existing API and behavioral contracts

### Requested Design Direction
Apply the user's recommended design passes during implementation:
- `/typeset`
  tighten labels, reduce decorative text noise, and make icon-adjacent copy feel more deliberate
- `/colorize`
  use semantic, tinted icon treatments instead of colorful emoji bursts
- `/layout`
  normalize icon placement, spacing, and supporting metadata rows so icons read as interface signage rather than decoration
- `/animate`
  add only restrained motion: hover/focus acknowledgment, state transitions, and progress feedback that respect `prefers-reduced-motion`

### Scope
In scope:
- dashboard pages and tabs for patient, caregiver, clinician, and admin
- shared dashboard-adjacent modules used inside those flows
  - `NotificationBell`
  - `NotificationCenter`

Out of scope for this phase unless explicitly expanded later:
- auth, signup, and onboarding screens
- non-dashboard utility/test files

### High-Risk Findings
- `PatientHEPTab.tsx` and `ClinicianWorklistTab.tsx` currently infer message styling from `message.includes("✅")` and similar string checks.
  Replacing emoji there is not a search/replace task; it requires formalizing message state so visuals are decoupled from string content.
- `NotificationBell.tsx` and `NotificationCenter.tsx` duplicate notification-type icon mapping.
  If left separate, icon drift will return quickly.
- Patient and clinician HEP/worklist surfaces are still legacy, inline-style-heavy modules.
  Emoji removal should not happen without a small structural cleanup because icon placement, spacing, and copy are intertwined.
- Rating experiences in caregiver/admin currently use text glyphs.
  These should become icon buttons/displays without changing the numeric payloads.

### Icon System Direction
- Canonical icon family: `lucide-react`
  This is already used widely across dashboards and matches the requested clean, sleek direction.
- Prefer signage over decoration:
  - status confirmation: `CircleCheck`, `BadgeCheck`, `CheckCircle2`
  - denial/cancel/error: `CircleX`, `Ban`, `TriangleAlert`
  - reminders/schedule: `Clock3`, `AlarmClock`, `CalendarDays`
  - care plan/prep: `ClipboardList`, `NotebookPen`, `ListChecks`
  - messaging: `MessageSquare`, `MessagesSquare`
  - general notifications: `Bell`, `BellRing`
  - exercises: `Dumbbell`, `Activity`
  - ratings: `Star` icons, not text stars
- Avoid inline emoji in labels, button copy, toast-like messages, and empty states.
- Avoid mixing raw glyph symbols and icon components in the same surface.

### Design Direction (Platform-Level)

#### Visual personality
- Admin surfaces: cinematic mission-control for healthcare operations (precise, serious, high-trust, high-information).
- Patient surfaces: calm, guided, confidence-building, less operationally dense than admin.
- Login: premium brand-entry surface that feels secure and modern without over-stylization.

#### Core visual system recommendations
1. Introduce explicit design tokens for color, surface elevation, spacing, type, radius, and motion.
2. Unify card language across admin and patient dashboards:
   - consistent panel border/inner highlight/elevation
   - consistent card header anatomy (title, micro-metadata, action slot)
3. Standardize semantic tone tokens:
   - success, warning, danger, info, neutral
   - each with icon tint + background tint + border tint variants
4. Consolidate chart visual language:
   - axis/legend/tooltip styles
   - consistent color ramps per metric category
5. Replace one-off inline styles in target screens with scoped classes and token-driven values.

#### Typography recommendations
- Heading family: `Manrope` or `Plus Jakarta Sans` (high legibility, premium but neutral).
- Body/UI family: `Source Sans 3` or `IBM Plex Sans` (dense-data readability).
- Numeric/KPI accent: optional tabular numerals via `font-variant-numeric: tabular-nums`.
- Establish a consistent hierarchy:
  - H1 screen title
  - H2 section title
  - H3 card title
  - label/microcopy/status caption scales

#### Spacing recommendations
- Adopt an 8px base grid with constrained increments (4/8/12/16/24/32/40).
- Standardize section rhythm:
  - 24px between major sections
  - 16px between cards in same group
  - 12px for internal card row spacing

#### Motion recommendations
- Use restrained, purposeful motion only:
  - panel enter: 220-320ms
  - hover/focus transitions: 140-180ms
  - data-state transitions: 180-240ms
- Keep motion linked to comprehension:
  - tab/screen transition continuity
  - status updates and queue changes
  - chart load skeleton to chart reveal
- Preserve no-motion alternatives via `prefers-reduced-motion`.
