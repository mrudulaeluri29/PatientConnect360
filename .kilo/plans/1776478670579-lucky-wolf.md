# Premium UI/UX Redesign Plan (Admin + Login + Patient Overview)

## Goal
Deliver a high-end redesign blueprint for six target screens with a consistent premium identity: richer shading, refined color usage, cleaner hierarchy, smooth animations, stronger chart readability, and a trustworthy clinical-operational tone.

## Scope
- Screen 1: Admin System Messages
- Screen 2: Admin Reports
- Screen 3: Admin Settings
- Screen 4: Admin Overview
- Screen 5: Login
- Screen 6: Patient Overview

## Cross-Screen Design Foundation

### Visual Style and Color Direction (Platform-Wide)
- Keep the existing premium healthcare-command direction, but reduce visual drift by standardizing one neutral base and one accent strategy per role.
- Use layered neutrals for depth: base canvas, panel surface, raised cards, and soft inset highlights.
- Keep admin accent in the existing violet family and patient accent in the calm violet-blue family, but reduce over-saturation for better credibility.
- Use semantic colors with tint + foreground pairs for readability:
  - success (green), warning (amber), danger (rose), info (blue), neutral (slate)
- Avoid random gradient variance between sections; gradients should be subtle and role-specific.

### Layout Improvements (Platform-Wide)
- Establish a predictable rhythm for all six screens:
  1. screen header/hero
  2. summary strip (KPIs/context)
  3. primary working surface
  4. secondary detail panels
- Use consistent section spacing: 24px major gaps, 16px card gaps, 12px internal stack gaps.
- Keep toolbar/filter areas sticky where list-heavy interaction is expected (messages, reports filters).

### Card, Graph, and Panel Treatment (Platform-Wide)
- Use one premium card pattern:
  - radius 14-18px
  - thin soft border
  - gentle shadow + subtle top highlight
  - clear header anatomy (kicker, title, support line, action slot)
- Apply semantic tint backgrounds at low opacity only; no heavy full-surface color fills unless emphasizing critical states.
- Normalize chart container styling (title, subtitle, legend, tooltip shell, axis labels, empty state).

### Motion and Animation System (Platform-Wide)
- Motion targets:
  - page/section enter: 220-320ms
  - hover/focus transitions: 120-180ms
  - chart/data transitions: 180-260ms
- Prefer transform/opacity animations for smooth 60fps behavior.
- Reserve stronger motion for comprehension moments only (tab switch, chart reveal, modal transitions).
- Reduced-motion mode: disable transform choreography, keep only instant state swap and minimal opacity.

### Typography and Spacing (Platform-Wide)
- Keep current premium stack (`Sora` for headings, `Public Sans` for body) and enforce hierarchy consistently.
- Enable tabular numerals for KPI and chart values.
- Tighten heading spacing and increase line-height for dense data cells.
- Standardize readable line lengths for subtitles and helper text.

### Structural Cleanup Requirement (Before Styling Passes)
- Consolidate duplicated CSS in `AdminDashboard.css` (messages and shared classes are currently defined in multiple conflicting blocks).
- Reduce leftover inline styles in target surfaces and replace with named utility/component classes.
- Introduce a small token extension layer in shell/page CSS for elevation, semantic tones, and chart colors.

## Incremental Plan: Group Similar Cancellation Reasons In Admin Analytics

### Goal
- Group semantically equivalent free-text cancellation reasons into clean reporting buckets for the admin cancellation pie/donut charts.
- Preserve the original raw cancellation reason text everywhere it is used for audit, notifications, and detailed operational review.

### Recommendation
- Implement grouping at the server analytics layer first.
- Do not change the DB schema or the cancellation input UX in the first pass.
- Keep the existing API contract shape unchanged (`{ reason, count }[]`), but return canonical grouped labels instead of raw free-text variants.

### Relevant Existing Files and Functions

#### Cancellation reason capture and persistence
- Patient cancel flow: `Client/web/src/pages/patient/PatientDashboard.tsx`
  - `handleCancel(...)`
- Caregiver cancel flow: `Client/web/src/pages/caregiver/CaregiverDashboard.tsx`
  - `handleCancel(...)`
- Shared visit API client: `Client/web/src/api/visits.ts`
  - `updateVisitStatus(id, status, cancelReason?)`
- Write-path route: `Server/src/routes/visits.ts`
  - `router.patch("/:id", ...)`
  - current behavior stores trimmed raw `cancelReason`
- Visit persistence model: `Server/prisma/schema.prisma`
  - `Visit.cancelReason`

#### Analytics aggregation and typing
- Analytics builder: `Server/src/lib/adminKpis.ts`
  - `buildAdminAnalytics(...)`
  - current aggregation key: raw trimmed `visit.cancelReason`
- Admin analytics API route: `Server/src/routes/admin.ts`
  - `GET /api/admin/analytics`
- Client analytics types: `Client/web/src/api/admin.ts`
  - `AdminAnalytics.charts.cancellationReasons`

#### Admin chart consumers
- Overview chart: `Client/web/src/components/admin/AdminOverviewPanel.tsx`
  - `Cancellation Reason Share`
- Reports chart: `Client/web/src/pages/admin/AdminDashboard.tsx`
  - `Cancellation Reason Breakdown`

#### Existing validation coverage
- Admin analytics integration test: `Server/src/__tests__/feature5.adminOverview.test.ts`

### Current Behavior Summary
- Raw reasons are captured as free text.
- Analytics currently buckets exact trimmed strings only.
- Visual shortening (`compactReasonLabel(...)`) only truncates display text; it does not merge categories.
- Result: minor text differences such as casing, punctuation, wording, and typos create separate slices.

### Implementation Strategy

#### Phase 1: Introduce canonical analytics grouping (recommended)
1. Add a dedicated server-side normalization helper.
   - Recommended new helper file: `Server/src/lib/cancellationReasons.ts`
   - Responsibilities:
     - normalize case, whitespace, apostrophes, and punctuation
     - classify raw strings into a deterministic reporting category
     - return a canonical label safe for charts and summaries

2. Keep raw text intact on the Visit record.
   - Do not overwrite `Visit.cancelReason`.
   - Do not change notifications or audit-log metadata to use grouped labels.
   - Do not change admin queue/detail tables that may still benefit from raw wording.

3. Update admin analytics aggregation only.
   - Modify `Server/src/lib/adminKpis.ts` inside `buildAdminAnalytics(...)`.
   - Replace the current raw-string aggregation key with the canonical category from the helper.
   - Continue returning `charts.cancellationReasons` as `{ reason, count }[]`.

4. Keep client chart code largely unchanged.
   - `AdminOverviewPanel.tsx` and `AdminDashboard.tsx` should continue consuming the same array shape.
   - Verify legend, tooltip, top-driver summary, and compact label behavior still read well with canonical labels.

#### Phase 2: Optional persistence upgrade (defer unless explicitly wanted)
- If the product wants grouped categories to become a first-class reporting dimension across future dashboards and exports, add a persisted field later, such as:
  - `cancelReasonCategory`
  - or `cancelReasonNormalized`
- This would require:
  - schema change
  - migration/backfill strategy
  - route write-path update
  - historical data handling rules
- This is not necessary for solving the current chart problem.

### Proposed Canonical Bucket Set
- `Patient not feeling well`
- `Unable to attend`
- `Scheduling conflict`
- `Transportation issue`
- `Family or personal issue`
- `Weather`
- `Other`
- `Unspecified`

### Initial Mapping Guidance
- Variants of inability to attend should group together:
  - `cant make it`
  - `can't make it`
  - `cannot make it`
  - `i cannot make it`
  - `unable to make it`
  - `not available`
  - `Unable to attend`
  - Suggested bucket: `Unable to attend`
- Health-condition wording should group together:
  - `not feeling well`
  - `feeling sick`
  - `unwell`
  - `patient sick`
  - Suggested bucket: `Patient not feeling well`
- Ambiguous/unrecognized free text should fall back safely:
  - typos, noise, low-signal text, one-off phrases
  - Suggested bucket: `Other`
- Missing or blank values should remain:
  - `Unspecified`

### Normalization Rules
- Lowercase before matching.
- Trim leading/trailing whitespace.
- Collapse repeated spaces.
- Normalize apostrophes and punctuation.
- Match by curated phrase rules first; avoid aggressive fuzzy matching in v1.
- Prefer deterministic mapping over edit-distance or AI-style inference.

### Risks and Tradeoffs
- False merges:
  - overly broad rules may combine distinct causes that ops teams want separated.
- Historical consistency:
  - changing rules later will change chart output for the same raw data unless categories are persisted.
- Over-fragmentation fallback:
  - too many buckets defeats the purpose; keep the taxonomy intentionally small.
- Under-classification:
  - some phrases may drop into `Other` until the mapping table is refined.

### Acceptance Criteria For This Increment

- Raw cancellation reasons remain preserved in the Visit record and unchanged in detailed flows.

### Out of Scope For This Increment

- Backfilling historical grouped categories into the DB.
- Changing notification/audit content from raw text to grouped labels.
- Introducing fuzzy or ML-based semantic clustering.
